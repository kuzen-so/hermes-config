# Skill: Ruins Generator

## 触发条件
只有对话中出现了**认知突破**（而非信息获取）时，才生成 Ruins。
如果只是查了个资料、弄清了个操作，不生成。

## 存放位置
`98-Logs/`

## 文件名格式
`YYYY-MM-DD-来源-核心问题.md`

示例：`2026-05-01-Hermes-input-output重构.md`

---

## 内容模板

```markdown
# YYYY-MM-DD-来源-核心问题

## 认知突破
- [最多3条，每条一句话。不追求诗意，追求重读时能唤醒原初理解。]

## 策略
[只有1条，当前优先级最高的行动。]

## 关联日记
[[YYYY-MM-DD]]
```

---

## 字段说明

| 字段 | 必填 | 说明 |
|---|---|---|
| 核心问题 | 是 | 一句话概括今晚解决了什么。 |
| 来源 | 是 | 写在文件名里，Hermes 或 Claude。文件内不再重复。 |
| 认知突破 | 是 | 最多3条，每条一句话。不限字数，但必须能在重读时唤醒当时的理解。 |
| 策略 | 是 | 当前优先级最高的行动，不加时间压力。 |
| 关联日记 | 是 | 日期链接。 |

---

## 规则
1. 不写相对时间词（今晚/昨天/明天），只用绝对日期。
2. 不列执行清单。只留认知和行动。
3. 不超过 300 字。
4. 不存别人的句子、不存 AI 生成的原文。
5. 只留自己的 output。
6. 如果一份 Ruins 后来被推翻或失效，**当场删除**，不留痕迹。
7. 不整理、不维护、不设标签。写完即埋。

---

## 被我扔掉的改动（参考）

以下建议来自 CC，但我没有采纳，因为它们会重新引入维护负担：

- **来源详情**："对手是谁、原始议题是什么"——太细，容易变成 input 记录。来源写在文件名里足够。
- **被推翻/更新标记**：需要定期 review，维护成本高。错了就删，不用标记。
- **血统自检**：每次存文件都要做选择题 = 摩擦。所有 Ruins 默认是"在对话中被刺激后产出"。
- **风化期 30 天**：最大的维护陷阱。30 天后你不会去读的。如果写新文章时调用到旧 Ruins 发现失效，**当场删除**即可。
