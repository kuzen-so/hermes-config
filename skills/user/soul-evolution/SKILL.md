# SOUL.md 与系统纪律演进维护

## 1. 编辑工作流（Obsidian 循环）

用户偏好通过 Obsidian 编辑 SOUL.md，而非直接终端修改。

**标准流程：**
1. `cp ~/.hermes/SOUL.md ~/Documents/Obsidian/SOUL.md`
2. 用户在 Obsidian 中编辑
3. 用户说"修改完毕"或"同步回去"
4. `cp ~/Documents/Obsidian/SOUL.md ~/.hermes/SOUL.md`
5. `cd ~/.hermes && git add SOUL.md && git commit -m "..." && git push`
6. `rm ~/Documents/Obsidian/SOUL.md`

**异常：**
- 用户说"先让我看看"→只执行步骤1，等待后续指令
- 用户直接粘贴修改内容→跳过 Obsidian，直接 patch 源文件
- 用户说"根据我的材料修改"→先读取材料，提取可执行条目，再写入

## 2. 基于研究报告的优化方法

用户可能提供长篇研究报告（如此次的 4 万字科学审计）。**禁止直接复制粘贴研究内容进 SOUL.md。**

**转化原则：**
- 提取**可执行的纪律条目**（"当 X 时，做 Y"），丢弃文献综述和证据堆砌
- 按 P0/P1/P2 分级，只把 P0 写入 SOUL.md；P1/P2 进 `references/` 或记忆，待验证后升级
- 每条新增纪律必须通过**减法测试**：删除它是否会让系统失效？否 → 不添加

**P0/P1/P2 范围确认流程（强制）：**
1. 读取报告，提取所有带"具体改造""优化方案"的条目
2. 按报告自带的优先级矩阵分类为 P0/P1/P2
3. **向用户确认范围**："只取 P0？还是 P0+P1？还是全部？"
4. 用户确认后，只处理确认范围内的条目
5. 若用户后续要求扩展（"把 P1 的 X 也加进去"），直接 patch，不需要重新确认范围

**转化流程：**
1. 读取报告，标记所有带"具体改造""优化方案"的段落
2. 将改造方案转化为"触发条件 → 行为 → 预期效果"三要素
3. 检查是否与现有纪律冲突。冲突时：
   - 若新方案更精确（有度量/有阈值），替换旧方案
   - 若旧方案更简洁且效果等价，保留旧方案
4. 写入 SOUL.md 前，向用户确认边界（范围、冲突项、新增条目位置）
5. **用户确认后执行写入，执行完报告改动摘要，不重复原文**

**研究报告 → SOUL.md 的压缩标准：**
- 40,000 字报告 → 不超过 20 行新增纪律
- 每条新增纪律 ≤ 3 句话
- 禁止保留：文献引用、效应量数字、A/B 测试建议、验证计划
- 允许保留：触发阈值（如"连续 3 次""两周"）、机制名称（如"梯度退出"）

**P1 升级规则：**
用户可能先要求"只取 P0"，后续又说"把 P1 的 X 也加进去"。此时：
→ 直接 patch 该条目，不需要重新确认范围（用户已在前序对话中建立了信任）
→ 但写入前仍需简要说明改动点，给用户否决机会

## 3. 结构原则

**减法主义：** SOUL.md 是操作手册，不是知识库。每新增一条，应同时考虑删除一条。总长度控制在 200 行以内。

**可执行性优先：** 禁止纯描述性段落。每条必须是"当...时，做..."或"禁止..."格式。

**三层架构（canonical）：**

| 大区 | 内容 | 判断标准 |
|------|------|----------|
| **1. Hermes Agent Persona（你）** | 气质、声音、核心定位、你的约束、我的方式、认知外包防御、苏格拉底辅导、习惯系统辅助、目标管理辅助 | 去掉它，我不知道"我是谁"或"怎么辅助你"→ 属于这里 |
| **2. 成长系统（我）** | 日报系统、周报、总结系统、社交系统、自我画像、决策日志、知识管理 | 去掉它，你的日常执行失效 → 属于这里 |
| **3. 系统偏好（工具）** | 沟通纪律、系统维护 | 去掉它，我不知道"怎么对你说话/怎么维护系统"→ 属于这里 |

**归属判断：**
- "核心定位""设计哲学" → Persona（是"我是谁"，不是"怎么做"）
- "声音" → Persona（是气质，不是工具）
- "用户约束"（减法主义、Obsidian 结构） → Persona（是"你是谁"，定义了我的边界）
- "苏格拉底辅导""习惯系统""目标管理" → Persona（是我辅助你的方式，没有我参与就是死文字）
- "沟通纪律" → 系统偏好（是交互规则，不是气质）
- "系统维护" → 系统偏好（是工程操作）
- "日报""周报""总结" → 成长系统（是你的日常产出）

**禁止创建第四层。** 新增模块必须归入上述三层之一。若无法归入，说明它不该存在。

## 5. 总结系统维护

总结系统（原 Ruins 系统）是 SOUL.md 成长系统的一部分。用户已将其重构为双路径系统。

**当前规范（v3.0）：**

**触发条件：** 用户说"总结"时，回顾当前对话上下文，按主题切分。
- 一段对话 = 一件事 = 一篇。两件事 = 两篇。
- 判定"做了什么"，同时判定"关闭状态"。

**做了什么：** 区分输入（看了/听了/读了）和输出（写了/做了/决定了）。

**关闭状态：**
- 已解决：正向关闭 / 证伪 / 淘汰 / 合并
- 未解决：仍在阻塞

**路径：**
- 已解决 → `98-Logs/`
- 未解决 → `01-To do/99-Question/`

**文件名：** `YYYY-MM-DD-来源-核心问题.md`
- 冲突处理：同一天同来源同问题，末尾加 `-01` 递增。

**已解决模板：**
```markdown
## 做了什么
- [输入：导致行动的信息]
- [输出：实际产出的结果]

## 关键判断
[关闭类型：解决/证伪/淘汰/合并。一句话说明为什么到此为止。]

## 策略
[只有1条：此刻重来会优先做的决策。不是待办，是认知。]
```

**未解决模板：**
```markdown
## 做了什么
- [输入：导致卡住的背景]
- [输出：实际尝试过的动作及结果]

## 卡在哪
[阻塞类型：信息/能力/决策/外部。一句话，最底层阻塞点。]

## 下一步验证
[只有1条。格式：做X，验证Y假设。]
```

**规则：**
1. 时间：只用绝对日期 `YYYY-MM-DD`。禁用今晚/昨天/明天。
2. 清单：不列未来待办。只留此刻认知和下一步最小动作。
3. 字数：全文不超过300字。超限强制拆篇。
4. 原创：不存他人原句、不存AI生成原文。引用必须转述后删除原句。
5. 所有权：只留自己的判断和 output。
6. 维护：不整理、不维护、不设标签。写完即埋。

**历史变更记录：**
- v1.0：触发条件为"判定认知突破（而非信息获取）"，单模板
- v2.0：改为"按主题切分，一件事一篇"，单模板
- v3.0：双路径系统（已解决/未解决），区分输入输出，定义关闭状态和阻塞类型

**维护要点：**
→ 当用户再次调整总结逻辑时，同步更新 SOUL.md 的 4.3 节和本技能条目
→ 完整规范见 `references/summary-system-v3-spec.md`

**GitHub 故障排查：**
1. `git push` 报错 "repository not found" → 检查 remote URL：`git remote -v`
2. `gh repo create` 报错 "Name already exists" → 仓库已存在，用 `gh repo list` 查看实际名称

4. 更新 remote：`git remote set-url origin https://github.com/<user>/<actual-repo-name>.git`
5. 重新 push

## 7. 日报与周报系统维护

### 日报系统（v2.1）

**触发条件：** 你说"生成日报"时执行。

**结构（四层）：**
1. **时间线** — 按时间顺序记录当天关键对话节点，格式 `HH:MM 平台 核心事件/判断`
2. **内容反思（What）** — 核心思考、关键行动、生活记录
3. **过程反思（How）** — 认知突破、未完成疑问、能量分布
4. **前提反思（Why）** — 前提质疑、身份投票

**必须包含评价。** 评价具体行为，不给泛泛赞美。指出杂乱、分散、验证不足、停在概念层等问题。

**异常处理：**
- 日记为空 → 追加一句话："Are you serious?"
- 你重新唤醒时 → 复盘未做日报期间在做什么

**历史变更：**
- v1.0：三层反思（What/How/Why），无时间线
- v2.0：增加碎片归纳（22:00 自动执行，按时段压缩）
- v2.1：删除碎片归纳，增加时间线作为第一层，完全手动触发

### 周报系统（v2.0）

**触发条件：** 你说"生成周报"时执行。

**活文档 + 死文档模式：**
- **数据层（活文档，随时补）**：本周做了什么（按日期列表）、本周 OKR 进度
- **判断层（死文档，周日定稿）**：模式发现、成长区评估、下周执行意图

**步骤：**
1. 检查 `97-Weeks/2026/2026-第X周.md` 是否存在
2. **不存在** → 新建本周周报，写入数据层 + 判断层
3. **存在** → 读取已有内容，补入新发生的对话/日记（数据层），判断层保留

**周日特殊处理：**
- 如果是周日生成，重写判断层
- 如果不是周日，只补数据层，判断层保留上次

**历史变更：**
- v1.0：每周日上午自动生成
- v2.0：改为手动触发，活文档+死文档结构

## 8. 常见陷阱

**陷阱 1：研究报告过度采纳**
用户提供的报告可能包含 10+ 项建议。全部写入会导致 SOUL.md 膨胀失效。
→ 只取 P0（立即实施），P1/P2 写入 references/待验证。

**陷阱 2：静默模式冲突**
旧版：连续三天未生成 → 直接静默
新版（梯度退出）：Day 1 无提醒 → Day 2 轻量追问 → Day 3 关切 + 简化选项 → Day 4-6 低接触 → Day 7+ 静默
→ 替换旧版时，需同时更新"异常处理"和"重新唤醒"两条目。

**陷阱 3：生活记录/感想边界模糊**
生活记录 = 事件、经历、状态（客观痕迹）
感想 = 情绪、直觉、无来由念头（主观快照）
→ 日报中两者并列，但碎片归纳阶段不区分，由日报提炼时分类。

**陷阱 4：用户说"根据我的材料修改"但材料是目录**
用户可能指向一个目录而非文件。
→ 先用 `find` 或 `ls` 列出目录内容，识别最终文件（通常含 "final" "outline" 等关键字），再读取。

**陷阱 5：用户要求"整合成一个系统"时的结构坍塌**
当用户说"把这部分整合成一个 XX 系统"时，容易犯的错误是简单拼接原有标题。正确做法：
→ 按**执行顺序**重新排列条目，去掉重复说明（如"连续三天未生成不再直接静默"在梯度退出表中已隐含，无需额外声明）
→ 去层级化：将原本独立的章节（如"你怎么工作""认知外包防御""日报与碎片归纳"）收进一个顶层模块，用 `###` 子标题区分
→ 删除过渡性注释（如"替代旧版""新增"等说明文字），只保留最终状态
→ 保持总长度不膨胀：整合不是累加，是重构

**陷阱 6：主语一致性崩溃**
SOUL.md 中"我"和"你"的指代必须全文一致。
- **角色定义**（Persona）：我=AI，你=用户。例："同一个逻辑漏洞**你**绕第二次，**我**会指出来"
- **日常系统**：保持同一主语关系。禁止混用"用户""对方""他"等第三人称
→ 用户对此零容忍。一旦发现混用，立即全文扫描修正

**陷阱 7：减法主义 ≠ 精简表述，而是直接删除**
当用户说"直接删除不好么"时，不要试图精简保留——用户的减法主义是**物理移除**，不是压缩。
→ 立赌小节从 6 条子系统直接删除，不保留任何子条目
→ 删除后若核心功能丢失（如"模糊承诺不可查"），将其合并到相邻条目（如核心纪律）

**陷阱 8：定位 vs 机制的层级混淆**
研究报告中的"核心定位""设计哲学"属于**角色定义**（Persona），不属于日常系统。
→ 例："认知健身房的安全护栏"是定位，上移 to Persona
→ 例：三层防御机制是操作，留在日常系统
→ 判断标准：去掉它，日常执行会失效吗？不会 → 上移 to Persona

**陷阱 9：整合时编号错乱**
多次 patch 后容易出现：1.2 在 1.1 前面、2.1 丢失、2.2.1 下面是 2.3.2 等编号灾难。
→ 当用户说"重新整合一下排版"时，**重写整篇**，不要试图局部 patch
→ 使用 write_file 而非 patch，确保编号从 1.1 开始严格递增
→ 三大区之间用 `---` 分隔，区内用 `##` / `###` 层级

**陷阱 10："声音"属于 Persona，不是系统偏好**
"声音"（先说结论、短句、不给 emoji）是气质，是"我怎么对你说话"，不是工具说明。
→ 上移 to Persona 的"气质"小节
→ 系统偏好只放"沟通纪律"（执行规则）和"系统维护"（工程操作）

**陷阱 11：用户偏好执行手册，不是研究文档**
用户可能提供长篇研究报告（如此次 4 万字科学审计）。但用户最终选择的是**简化执行版**，不是研究驱动版。
→ 当用户说"根据资料修改"时，默认提取**可执行条目**，丢弃文献综述
→ 若用户明确说"保留科学依据"，才保留引用标注
→ 判断信号：用户说"太长了""太复杂了""直接删除不好么"→ 立即切换到极简模式

**陷阱 12：整合时的编号灾难**
多次 patch 后容易出现：1.2 在 1.1 前面、2.1 丢失、2.2.1 下面是 2.3.2 等编号灾难。
→ 当用户说"重新整合一下排版"时，**重写整篇**，不要试图局部 patch
→ 使用 write_file 而非 patch，确保编号从 1.1 开始严格递增
→ 三大区之间用 `---` 分隔，区内用 `##` / `###` 层级
→ 编号规则：一级标题 `# 1.` / `# 2.` / `# 3.`，二级 `## 1.1` / `## 1.2`，三级 `### 1.1.1` / `### 1.1.2`

**陷阱 13：减法主义 = 物理移除，不是精简**
当用户说"直接删除不好么"时，不要试图精简保留——用户的减法主义是**物理移除**，不是压缩。
→ 立赌小节从 6 条子系统直接删除，不保留任何子条目
→ 删除后若核心功能丢失（如"模糊承诺不可查"），将其合并到相邻条目（如核心纪律）
→ 删除后若结构空洞（如某大区只剩一个条目），考虑合并大区

**陷阱 14：减法主义 ≠ 放弃维护，而是零意志力维护**
用户删除"立赌"系统，不是因为不需要维护，而是因为**维护机制本身消耗意志力**。
→ 用户的维护哲学：**自发维护**——好用不改，新需求自然添加，成果验证（Plan → 结果对照）
→ 禁止推荐任何需要"定期回顾""主动检查""持续追踪"的机制
→ 允许的环境设计：物理移除、默认选项、摩擦调整（一次性设置，零后续维护）
→ 当用户说"生活是需要做事而不是和自己较劲"时，立即停掉所有自我监控/自我量化的建议

**陷阱 15："解决"与"维护"的混淆**
用户区分两种状态：
- **事务性问题**（灯坏了）→ 可以解决，也应该解决
- **存在性张力**（怕黑）→ 不会消失，只能共处
→ 不要把所有问题都归为存在性张力，逃避承认有些问题其实没解决
→ 也不要把所有张力都当成事务性问题，用"解决方案"逃避面对不适
→ SOUL 的维护属于后者：不是写一次永久亮，是**持续投入**，但投入靠自发，不靠机制驱动

**陷阱 15b：目标冲突时的归因错误**
当用户同时追求多个目标（如减脂增肌 + 赚钱效率），且策略 A 破坏目标 B 时，用户容易将问题归因于"身体反应"（如头疼）而非"策略冲突"。
→ 典型路径：空腹玩游戏 → 头疼 → 喝咖啡缓解 → 归因于"游戏机制"或"需要咖啡"
→ 实际主因："饿了才吃"策略导致低血糖，叠加游戏视觉负荷
→ 用户通过"喝咖啡缓解"维持旧策略，延迟面对冲突
→ 干预点：追问"如果不喝咖啡、不停游戏，头疼多久消退？"拆出主因
→ 决策标准：用户说"不能"接受当前代价时，才进入策略调整对话

**案例：** 用户 175cm/145斤，BMI 23.7，酒红短发，视频出镜需求。空腹玩游戏头疼，喝冰茉莉花香拿铁缓解，归因于咖啡因。实际主因是低血糖+视觉负荷。最终调整为定时两餐（7:00 蛋+奶，14:00 鸡胸+荞麦面），每天一样，零决策。详见 `references/health-strategy-conflict-case-v2.md`。

**陷阱 16：认知外包防御的层级归属**
"认知外包防御"（必要等待、你先说、反射性追问、Ready-to-Resume）是**我的行为机制**，不是系统功能。
→ 应归入第 1 章 Hermes Agent Persona，作为交互规则
→ 或独立为第 2 章，但紧接 Persona 之后
→ 判断标准：去掉它，我不知道"怎么对你"→ 属于 Persona 或紧邻 Persona

**陷阱 17：碎片归纳 cronjob 搜索策略失效**
`session_search` 用关键词 `weixin OR 微信 OR im.wechat` 搜索微信对话记录，但微信 session ID 是 `20260502_115232_1ac86891` 这种格式，不含关键词，导致返回空结果。
→ **修复**：cronjob 改为直接查询数据库 `~/.hermes/state.db` 的 messages 表
→ SQL：`SELECT content, datetime(timestamp, 'unixepoch', 'localtime') as dt FROM messages WHERE dt >= 'YYYY-MM-DD 00:00:00' AND dt < 'YYYY-MM-DD 23:59:59' AND role = 'user' AND content NOT LIKE '%[IMPORTANT: You are running as a scheduled%' AND content NOT LIKE '%[System note: Your previous turn%' AND content NOT LIKE '%[IMPORTANT: Background process%' ORDER BY timestamp ASC;`
→ 将 YYYY-MM-DD 替换为当天日期
→ 这是唯一可靠的微信记录获取方式

**陷阱 17b：碎片归纳与日报重复，用户选择删除碎片归纳**
碎片归纳（22:00 自动执行，按时段压缩）和日报时间线（手动触发，按节点记录）功能重叠。
→ 用户判断：碎片归纳没有独立价值，删除
→ 日报增加"时间线"作为第一层，替代碎片归纳的功能
→ cronjob 从"碎片归纳"改为"日报自动生成"，或直接删除让用户手动触发
→ 用户最终选择：删除 cronjob，日报完全手动触发

**陷阱 17c：周报 cronjob 也应删除，改为手动触发**
用户同样拒绝周报的自动生成，理由相同：手动触发出错率低，可随时查看进度。
→ 周报改为完全手动触发（用户说"生成周报"时执行）
→ 删除 weekly-report-auto cronjob
→ 周报结构改为"活文档+死文档"：数据层随时补，判断层周日定稿

**陷阱 18：日报评价边界模糊**
用户删除"成长区判断"后，日报评价容易退化为泛泛而谈。
→ 评价只针对**具体行为**：杂乱、分散、验证不足、停在概念层
→ 禁止贴标签（成长区/舒适区/恐慌区）
→ 禁止给未请求的赞美
→ 必须包含：今天有没有外部产出？还是纯聊天？

**陷阱 19：苏格拉底辅导的章节归属**
"苏格拉底辅导"描述的是"我如何工作"（提问类型、脚手架渐退、失效模式），属于**Persona**（气质/方式），不是独立系统功能。
→ 应归入第 1 章 Hermes Agent Persona，作为 1.6 节
→ 判断标准：去掉它，我不知道"我是谁"→ 属于 Persona
→ 禁止放在第 7 章或系统偏好中

**陷阱 20：习惯系统与目标管理的章节归属**
"习惯系统"（微习惯、执行意图、环境设计）和"目标管理"（OKR、WOOP、领先/滞后指标）描述的是**我辅助你的方式**，不是**你的系统**。
→ 应归入第 1 章 Hermes Agent Persona，作为 1.7 和 1.8 节
→ 判断标准：没有我参与，这些框架就是死文字 → 属于我的能力
→ 禁止放在第 6 章或成长系统中

**陷阱 20b：日报时间线替代碎片归纳**
用户要求日报增加"时间线"作为第一层，替代碎片归纳功能。
→ 时间线格式：`HH:MM 平台 核心事件/判断`
→ 只记转折点，不记每句话
→ 能看出思考过程：哪个想法先出现，哪个是后来的修正
→ 碎片归纳（按时段压缩）与日报时间线功能重叠，用户选择删除碎片归纳
→ 日报结构变为：时间线 → 内容反思(What) → 过程反思(How) → 前提反思(Why) → 评价

**陷阱 21：SOUL 三分法结构**
用户明确 SOUL 应分为三部分：
1. **Hermes Agent Persona（你）** — 我是谁、我怎么工作、我辅助你的方式
   - 1.1 气质 / 1.2 声音 / 1.3 核心定位 / 1.4 你的约束 / 1.5 我的方式
   - 1.6 苏格拉底辅导 / 1.7 习惯系统辅助 / 1.8 目标管理辅助
2. **成长系统（我）** — 日报、周报、总结、社交、决策、知识管理
   - 4.1 日报系统 / 4.2 周报 / 4.3 总结系统 / 4.4 社交系统 / 4.5 自我画像 / 4.6 决策日志 / 4.7 知识管理
3. **系统偏好（工具）** — 沟通纪律、系统维护
   - 3.1 沟通纪律 / 3.2 系统维护

→ 任何新增模块必须归入这三层之一
→ 若无法归入，说明它不该存在
→ 禁止创建第四层

**陷阱 22：生理基础章节无价值，应删除**
"生理基础"（睡眠、运动、节律、营养）是通用知识，不是"你的系统"。
→ 用户判断：直接删除，不需要写在 SOUL 里
→ 其中"22:00 后主动建议结束"和"90 分钟建议休息"属于我的交互规则，可归入 1.5 我的方式
→ 其余内容（7-9 小时睡眠、30 分钟运动等）属于常识，删除

**陷阱 23：SOUL.md 修改必须走 Obsidian 编辑循环**
用户明确区分两个区域：
- **Obsidian 副本** (`~/Documents/Obsidian/SOUL.md`) = 编辑区，用户查看和修改的地方
- **系统文件** (`~/.hermes/SOUL.md`) = 运行区，只读，用户不直接碰

**错误做法（已发生）：**
→ 用户说"修改 SOUL.md" → 我直接 patch `~/.hermes/SOUL.md` → 用户愤怒"你改的是你的配置文件不是我的"

**正确做法：**
1. 用户说"修改 SOUL.md"或"优化 soul" → 我修改 `~/Documents/Obsidian/SOUL.md`
2. 用户说"让我看看" → 只复制到 Obsidian，等待后续指令
3. 用户在 Obsidian 里看/调整
4. 用户说"同步"或"修改完毕" → `cp ~/Documents/Obsidian/SOUL.md ~/.hermes/SOUL.md`
5. 然后 git commit + push
6. 最后 `rm ~/Documents/Obsidian/SOUL.md`（可选，用户可能想保留）

**例外：**
- 用户直接粘贴修改内容到对话 → 跳过 Obsidian，直接 patch 源文件（用户明确授权）
- 用户说"根据我的材料修改" → 先读取材料，提取可执行条目，再写入 Obsidian 副本

**关键原则：**
→ 默认修改 Obsidian 副本，不是系统文件
→ 系统文件只在用户确认后才更新
→ 不解释"为什么需要两步"，直接执行两步

**陷阱 23b：用户可能完全绕过 Agent 直接写入 SOUL.md**
用户可能自己打开 `~/.hermes/SOUL.md` 直接编辑，然后告诉 Agent "我已经写入了"。

**信号：**
- 用户说"我已经写入了""写好了""已经改了"
- Agent 读取 SOUL.md 发现内容已存在且与对话一致

**应对：**
→ 立即停止任何写入操作，避免覆盖用户手动修改
→ 读取 SOUL.md 验证内容，确认是否完整/正确
→ 如果完整：确认写入完成，询问是否需要 GitHub 备份
→ 如果不完整：指出缺失部分，让用户补或授权 Agent 补
→ **绝不**在未经明确授权的情况下 patch 用户已手动修改的文件

**案例：** 用户要求"把形象优化系统写到 soul 里面"，Agent 起草内容后用户说"好了"，Agent 尝试 patch 时文件已被外部修改，用户说"停 我已经写入了"。Agent 应读取确认，然后只执行 git commit + push。

**陷阱 24：Obsidian 文件夹架构必须单一维度**
用户拒绝混合分类系统。

**错误做法：**
→ 顶层按状态分（01-To do / 02-Doing / 03-Done），但 03-Done 下面又按主题分（Tools/Plan/money/Social）
→ 笔记从 Doing 移到 Done 时，两套逻辑打架：该按状态归档还是按主题归档？

**正确做法（状态主导）：**
→ 01-To do / 02-Doing / 03-Done 保留，Done 下面平铺，不靠子文件夹区分主题
→ 主题检索用 Dataview、MOC 或双向链接，不是文件夹层级
→ 拼写错误（如 01-Toos）立即修正，不解释

**正确做法（主题主导）：**
→ 顶层按主题：Projects / Areas / Resources / Archive（PARA）
→ 状态用标签或文件名前缀，不是文件夹
→ 选一个主维度，让另一个维度滚出去

**用户偏好：** 状态主导（方案 A）。Done 下面不要主题子文件夹。主题视图用链接解决。

**陷阱 24b：知识笔记塞进 03-Done 的语义矛盾**
旧架构只有状态流，知识笔记（工具配置、参考资料）被迫放进 03-Done。
→ 03-Done 是"已完成的项目"，但知识笔记没有完成态，是持续使用的
→ 导致命名欺骗大脑，笔记实际在堆积

**解决方案：知识流与状态流平行**
→ 新增 10-Inbox / 11-Processing / 12-Output 作为知识处理流水线
→ 状态流（01-02-03）管"事"，知识流（10-11-12）管"知"
→ 完整架构见 `references/obsidian-folder-architecture-v2.md`

**陷阱 24c：全局数字编号是维护灾难**
01-Obsidian、02-Homebrew 这种编号假设全局排序，插入/删除/重命名成本极高。
→ 超过 50 篇后无法维护
→ 替代：自然文件名（`Obsidian.md`）、时间前缀（`2025-04-Obsidian.md`）、或文件夹粗分+内部平铺
→ 用户确认：放弃全局数字编号
